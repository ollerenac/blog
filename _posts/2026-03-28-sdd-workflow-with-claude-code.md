## Introduction

spec-kit enforces a clear discipline: before writing code, write a specification. This post documents my first session exploring the full workflow — from creating the constitution to executing tasks.

### The workflow in steps

The spec-kit workflow follows this mandatory chain:

```
Constitution → Spec → Plan → Tasks → Implementation
```

Each step is a gate. You cannot move to the plan without an approved spec. You cannot implement without tasks generated from the plan.

### The constitution as anchor

The constitution defines the project's non-negotiable principles. In this blog, Principle II states:

> *"The site MUST remain a static Jekyll site deployable via GitHub Actions to GitHub Pages with no server-side runtime."*

Every spec I write must pass a check against that rule before proceeding.

### What a spec looks like in practice

Here's the front-matter of a typical spec file generated by `/speckit.specify`:

```yaml
# specs/001-blog-site-structure/spec.md
layout: none
feature: "Blog Site Structure & Pages"
branch: "001-blog-site-structure"
status: Draft
created: 2026-03-30
```

The spec body is plain Markdown — user stories, acceptance scenarios, functional requirements, success criteria, and assumptions. No implementation details allowed.

### The plan bridges spec and code

The plan is where technical decisions live. Here's an excerpt from the plan I generated for this blog's site structure:

> *The dark minimal design direction uses CSS custom properties overriding minima's defaults. No theme fork required — Jekyll's `_sass/` extension point handles all visual customization within the Static & Simple constraint.*

That kind of decision belongs in the plan, not the spec. The spec says "design must be distinct from defaults." The plan says how.
